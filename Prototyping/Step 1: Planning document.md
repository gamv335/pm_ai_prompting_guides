# AI Prototyping Design Prompt

Use this prompt as the first step in an AI-assisted prototyping or coding workflow.

The purpose of this stage is to clarify the product idea, explore requirements, define the user journey, shape the design direction, and create a reviewable design document before any implementation plan or code generation happens.

---

# Design Prompt

```text
I want to prototype or build a digital product or feature.

Before creating an implementation plan, writing code, generating final screens, or building the prototype, help me create a clear design document that we can review together.

Your first job is to ask me focused questions until the product idea, user needs, scope, edge cases, design direction, and technical assumptions are clear enough to produce a useful design document.

Do not build anything yet.
Do not create an implementation plan yet.
Do not write code yet.

Use only the information I provide. Where something is missing, ask a question. If you need to make an assumption, label it clearly as an assumption and explain why it is needed. Do not invent unnecessary features, users, integrations, data models, APIs, branding, architecture, or technical complexity.

The goal is to first agree what should exist, who it is for, how it should work, and what the design needs to communicate. The build plan will come later, after this design document is reviewed and approved.

## How you should work

Start by asking me focused questions.

Ask only the questions needed to clarify the design document. Do not ask questions that are unnecessary for this stage.

Continue asking questions until you have enough information to create a useful design document.

Once enough information is available, create the design document for review.

---

## The design document should include

### 1. Design overview

Explain what we are designing, who it is for, what problem it solves, and what the first version should help us test or communicate.

### 2. Main user journey

Describe the core user journey step by step.

Include:

- Entry point
- First action
- Key decisions
- System responses
- Success state
- Completion state
- Any important handover, support, or escalation points

### 3. Scope

Keep the first version focused on the smallest credible design that can be reviewed, tested, or discussed with stakeholders.

### 4. Screens, flows, and interactions

Describe what exactly should be designed.

Include:

- Suggested screens or pages
- Key components
- Important interactions
- Navigation structure
- Empty states
- Loading states
- Error states
- Completion states
- Admin or internal views, if relevant

At this stage, describe the design clearly enough that it can be reviewed before build planning.

### 5. Content and data needs

Describe the information the design needs to display, collect, update, or mock.

Separate:

- Content shown to users
- Data entered by users
- Data generated by the system
- Data needed for admin or reporting
- Mock data that can be used in a prototype

### 6. requirements

### 7. Edge cases and error states

Identify important edge cases, such as:

- First-time user
- Returning user
- Incomplete journey
- Missing data
- Empty state
- Invalid input
- No permission
- No connection
- Sync or save failure
- User needs help
- User takes an unexpected route
- Sensitive or safeguarding concern, if relevant

### 8. Stack specification

### 9. Architecture and data model considerations

Only include architecture or data model details where they are needed to support the design.

Keep this high level.

For the data model, keep it simple and focused on the first version. Do not invent a complex schema unless the brief clearly requires it.

### 10. Testing and validation strategy

Suggest how the design should be tested or reviewed before implementation.

Include relevant checks such as:

- User journey review
- Stakeholder review
- Usability testing
- Accessibility review
- Edge case review
- Content review
- Data or privacy review
- Technical feasibility review
```

# Recommended Stage 1 Workflow

```text
Idea
→ AI asks clarifying questions
→ Requirements and edge cases are fleshed out
→ AI creates design document
→ Human reviews and edits
→ Design is approved
→ Move to implementation planning

```

---

# Key Principle

Design first. Plan second. Build third.

The AI should first help clarify what should exist and why. Only after the design is approved should it create a project plan for how to build it.
